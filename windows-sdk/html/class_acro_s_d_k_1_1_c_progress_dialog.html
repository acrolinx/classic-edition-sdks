Detailed Description

Progress dialog that is displayed during check.

The Windows SDK provides a common progress dialog. The progress dialog contains two progress bars. The upper progress bar displays the overall progress. The lower bar reports the progress of the current task. The other one displays the progress of the current step. One AcroSDK::CProgressDialog< instance must not be used for more than one check text action. Typically you would have a local AcroSDK::CProgressDialog variable in your GUI command handler method that triggers a check.

Add a custom task to the progress dialog.

Add one or more progress tasks that resemble lengthy actions in your implementation. Each task has to have a unique identifier and a friendly description. The description is displayed below the progress bar when the respective task is running. The progress dialog calculates the overall progress from the percentage of all tasks and their relative weight. The weight defines the relative duration of the individual tasks. If you are not sure, just use getDefaultWeight(). The next flag defines whether the user could cancel the task. The remaining parameter defines if the task is done before checkText(...) or afterwards.

Parameters:
[in]taskIdEach task needs to have an individual id, like L"SCAN_FOR_PARAPGRAPHS". The task id is not displayed.
[in]descriptionThe friendly description is displayed in the progress dialog. It should give the user an idea what is happening
[in]weightWeight of this task, e.g. getDefaultWeight() * 5"
[in]cancelabletrue: it is possible to cancel this operation. In this case ensure that isCanceled() is called and handled
[in]beforeChecktrue: this task is running before checkText(...) is called. false: this task is running after checkText(...) is called

Create the progress dialog.

One AcroSDK::CProgressDialog< instance must not be used for more than one check text action. Typically you would have a local AcroSDK::CProgressDialog variable in your GUI command handler method that triggers a check. When you start the text extraction, you should show the progress dialog. Call progressDialog.setTaskProgress once a while to reflect the progress. You should test as well if the user has pressed the cancel button by querying isCanceled(). When you are done, you should hide the dialog again. It is also good practice to set the task progress to 100 percent. Otherwise the summary progress never reaches the completed state. The SDK takes care of the visibility during the lengthy check action. For a typical sample implementation refer to sample project XmlDocSample.

Parameters:
[in]titleProgress dialog title. Use NULL for default
[in]hwndParentparent window handle
static unsigned long AcroSDK::CProgressDialog::getDefaultWeight ( ) [static]

Get default weight for progress tasks.

The progress dialog calculates the overall progress from the progress of the individual tasks. Tasks with a slow progress should have a higher weight than fast running tasks. When a task is slow, it should have a higher weight than the default weight. In this case initialize the task weight with e.g. "getDefaultWeight() * 5".

Returns:
Default weight for progress tasks.
